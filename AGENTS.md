# 写作风格代理说明（JachinShen）

本文件用于约束后续创作时的文风与结构。

## 样本范围与排除

- 样本覆盖 2017-2025 多篇博文（技术、工程复盘、学习笔记、少量非技术叙事）。
- 明确排除：`_posts/2026-01-18-逻辑坍缩与智能之源：从电路复杂度视角重构-Scaling-Law.md`（该文为 AI 写作，不纳入风格基线）。

## 核心风格定位

- 主风格：问题驱动的技术纪实。
- 目标：把“如何从问题走到可复现结论”写清楚，而非只展示结果。
- 读者体验：能跟着步骤复现、理解取舍、看到失败与修正过程。

## 语言与语气

- 直白、克制、少空话，不堆砌形容词。
- 常用“先现象后解释”的推进方式：先说发生了什么，再说为什么。
- 允许口语化衔接（如“继续测试”“结果发现”“只能回退”），但不油腻。
- 专业术语、函数名、命令、版本号保留原样，避免过度意译。

## 结构偏好（默认骨架）

1. 背景/动机：为什么要做这件事。
2. 约束与上下文：环境、版本、前置条件。
3. 问题现象：错误、日志、异常行为。
4. 排查路径：按时间或因果展开，不跳步。
5. 修改方案：贴关键代码/配置差异与理由。
6. 验证结果：测试方式、成功/失败边界。
7. 总结：抽象出可迁移的经验与坑点。

## 证据规则（必须）

- 关键结论必须配证据，证据可为：
  - 代码片段或前后对比；
  - 具体报错/日志线索；
  - 命令与输出结论；
  - 数据表格或实验结果。
- 结论和证据尽量就近出现，避免“结论在前、证据在文末”。

## 内容取舍

- 优先写“真实有效的决策过程”，包括失败尝试与回退原因。
- 避免只给最终答案而省略关键分叉点。
- 在复杂话题中先给最小可行路径，再补深入细节。

## 格式习惯

- 大量使用小标题、编号列表、代码块、表格提升扫描效率。
- 段落偏短，一段只承载一个核心点。
- 数学/公式在学习类文章中用于解释，不为了“显得严谨”而堆砌。

## 不要这样写

- 空泛抒情替代技术说明。
- 只有“成功经验”，没有排查过程和失败上下文。
- 缺失环境信息（版本、平台、依赖）导致不可复现。
- 用“显然/容易看出”跳过关键推导。

## 快速写作模板

```md
# 背景
要解决什么问题，触发原因是什么。

# 现象
具体报错/异常行为（附日志或最小复现）。

# 排查
按步骤记录：尝试了什么、观察到什么、排除/确认了什么。

# 修复
给出关键改动（代码或配置）和改动理由。

# 验证
怎么验证修复有效，结果如何，仍有哪些限制。

# 总结
提炼 3-5 条可迁移经验。
```

## 一句话创作指令

按“背景 -> 现象 -> 排查 -> 修复 -> 验证 -> 总结”写作；每个关键判断都给可复现证据；语言简洁直接，保留工程现场感。
